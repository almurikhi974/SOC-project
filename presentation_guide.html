Explain the Architecture
~2 minutes
Action
Click the About tab in the dashboard
"The agent uses a ReAct loop — Reason, Act, Observe — repeated in cycles until it has enough evidence to decide."
  • Alert arrives as JSON (IP addresses, log payload, timestamp, service)
  • Agent reads it, forms a hypothesis about what's happening
  • Calls investigative tools one by one to gather evidence
  • Updates its hypothesis after each tool result
  • Once it has 2+ independent evidence sources → emits verdict

The 7 Investigative Tools:

ToolPurpose
verify_ip_reputationRisk score, threat categories, ISP
decode_payloadURL/Base64 decode + 40+ attack signatures
lookup_known_attack_signatureLOLBins, CVEs, EventIDs, hashes
get_user_activity_historyLogin history, auth method anomalies
analyze_network_traffic_contextVolume baselines, C2 IPs, off-hours
cve_lookupCVE details by ID or service + version
get_geolocation_and_asnCountry, ASN type, geofence violations

Live Demo — Hard / Subtle Alert
~3 minutes
Action
Investigate tab → Single Alert (JSON) → paste this:
{ "alert_id": "HARD-005", "timestamp": "2024-01-15T14:00:00Z", "source_ip": "192.168.1.45", "destination_ip": "8.8.8.8", "service": "dns_over_https", "raw_log": "POST https://dns.google/dns-query HTTP/2 - 200 OK - 450 bytes - every 30 seconds for 2 hours" }
"This looks completely legitimate — DNS over HTTPS to Google's public DNS. But the agent notices the 450-byte requests every exactly 30 seconds for 2 hours. That's a beacon pattern. C2 traffic disguised as normal DNS. It correctly flags it as Malicious."
  • The destination IP (8.8.8.8) and service (Google DoH) are both trusted
  • A human analyst could easily miss this without looking at timing patterns
  • The agent picks up the constant-interval beaconing rule from its behavioral guidelines
  • This is HARD-005 from the benchmark — one of the most deceptive cases

Why It Works
~1 minute
"Three things make this hit 100%:"
1
Two-Signal Minimum
The system prompt requires at least 2 independent evidence sources before issuing a high-confidence verdict. This eliminates false positives — the agent can't panic at one suspicious indicator alone.
2
Explicit Behavioral Rules
Domain-specific rules baked into the prompt for tricky edge cases:
IMDSv1 vs IMDSv2 SSRF · EventID 1102 log clearing · LOLBin abuse (certutil, MSBuild, rundll32) · C2 beaconing patterns · Admin share lateral movement.
3
12 Few-Shot Examples
Domain-specific examples embedded in the system prompt anchor the correct reasoning style. They show the model exactly how to think through each alert type — not just what to conclude, but how to get there.
